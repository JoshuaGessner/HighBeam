## Project Overview

HighBeam is a decentralized multiplayer framework for BeamNG.drive. It has three components:

1. **Launcher** (`launcher/`) — Lightweight Rust CLI that syncs mods, installs the client mod, launches BeamNG.drive, and exits
2. **Client mod** (`client/`) — Lua mod running inside BeamNG.drive
3. **Server binary** (`server/`) — Standalone Rust application

Key design principles:
- **No centralized authentication** — servers handle their own auth
- **No centralized infrastructure** — server discovery uses a P2P gossip mesh (v0.8.0), not a central relay
- **Lightweight launcher** — syncs mods and launches the game, then exits (not an always-running proxy)
- **Direct connection** — the in-game client mod connects directly to the server (no traffic through the launcher)
- **IP privacy** — server browser never displays IP addresses; IPs are resolved internally at connect time only
- **Fully documented protocol** — versioned, specified in PROTOCOL.md
- **Extensible** — both client and server support plugins

### Active Milestone: v0.8.0 — Community Node Discovery Mesh

The current implementation focus is v0.8.0: a decentralized P2P server discovery system. Key implementation details are in VERSION_PLAN.md. Quick reference for the four phases:

1. **Phase A (Server Node Core):** `server/src/community_node.rs` — state structs, HTTP listener on port 18862, gossip loop (30s interval), merge logic (newer `last_seen` wins), `community_node.json` persistence, input validation. Server ID format: `hb-` + 6 hex chars. Caps: 500 servers, 200 peers. Endpoints: `GET /servers` (no IPs), `GET /resolve/{id}` (rate-limited), `POST /gossip` (node-to-node). No new crate deps — uses existing `tokio`, `serde_json`, `reqwest`, `rand`.
2. **Phase B (Server GUI + Console):** "Community" tab in `gui.rs` with enable/disable checkbox, seed management, region/tags, live status. Console commands in `control.rs` (`community enable/disable/status/port/region/tags/add-seed/remove-seed`). Hot-start/stop without game server restart.
3. **Phase C (Client Browser):** Rewrite `browser.lua` Browse tab — replace `relayUrl` with `communityNodes` list, `fetchCommunityServers()` tries nodes in sequence, `resolveAndConnect(serverId)` resolves IP without displaying it. Favorites/recents by `server_id` not `host:port`. Show mods column, latency from HTTP round-trip.
4. **Phase D (Integration & Testing):** Two-node convergence test, client fetch→resolve→connect, rate limit verification, `/servers` IP-leak-free serialization test.

## Original Work Policy

> **All HighBeam code must be 100% original. Never reference, translate, port, or adapt BeamMP source code.**

BeamMP (Server, Launcher, Client Mod) is licensed under **AGPL-3.0**. HighBeam is **GPL-3.0-only**. These licenses are incompatible. Copying or translating BeamMP code — even rewriting their C++ in Rust, or mimicking their Lua patterns line-by-line — creates a derivative work and would force relicensing the entire project.

**Rules that MUST be followed:**

1. **Never open BeamMP source files while writing HighBeam code.** Not for reference, not for "inspiration", not to see how they did it.
2. **Never copy function signatures, packet formats, or wire protocols from their codebase.** Our protocol is independently designed and documented in PROTOCOL.md.
3. **Implement from first principles.** Use Rust/Lua documentation, general game networking references (Glenn Fiedler, Gabriel Gambetta), and BeamNG's own modding docs.
4. **If you need to understand a BeamNG API**, read BeamNG's own documentation or test in the game console. Do not look at how BeamMP calls it.
5. **`docs/reference/BEAMMP_RESEARCH.md` documents public-facing behavior for competitive analysis only.** It is NOT a code reference and must not be used to replicate BeamMP internals.
6. **Functional concepts are fine.** API names like `GetPlayers()`, directory conventions like `Resources/Server/`, and event names like `OnPlayerJoin` are generic functional concepts and are not copyrightable. But the implementation behind them must be our own.

## Language & Technology Standards

### Server (Rust)

- **Edition:** Rust 2021
- **Async runtime:** Tokio
- **Error handling:** Use `thiserror` for library errors, `anyhow` for application errors. Never `unwrap()` in production code paths — use `expect()` with a descriptive message only when failure is truly impossible (e.g., static regex compilation).
- **Logging:** Use `tracing` with structured fields. Always include player_id, vehicle_id, or session context where relevant.
- **Naming:** Follow Rust conventions — `snake_case` for functions/variables, `PascalCase` for types, `SCREAMING_SNAKE_CASE` for constants.
- **Dependencies:** Prefer well-maintained crates. Pin versions in `Cargo.toml`. Run `cargo audit` before releases.
- **Formatting:** Use `rustfmt` with default settings. Run `cargo fmt` before committing.
- **Linting:** Use `clippy` at `warn` level. Address all warnings before merging.
- **Testing:** Write unit tests for packet serialization, auth logic, and state management. Use `#[tokio::test]` for async tests.
- **Unsafe:** Avoid `unsafe` unless absolutely necessary. If required, document why and add safety comments.

### Client (Lua)

- **Runtime:** LuaJIT (provided by BeamNG.drive)
- **Style:** Follow BeamNG extension conventions:
  - Modules return a table `M`
  - Use `local` for all variables — no globals except the module table
  - `camelCase` for function and variable names
  - File-level functions are `local function name()` unless exported on `M`
- **Networking:** Always use non-blocking sockets (`settimeout(0)`)
- **Performance:** Keep `onUpdate` work under 1-2ms. Don't allocate tables in hot loops — reuse them.
- **Error handling:** Wrap socket operations in pcall. Never crash the game on a network error.

## Git Workflow

### Branching Strategy

```
main                    ← stable, tagged releases
├── develop             ← integration branch for next release
│   ├── feature/xxx     ← feature branches
│   ├── fix/xxx         ← bug fix branches
│   └── refactor/xxx    ← refactoring branches
└── hotfix/xxx          ← critical fixes branched from a tag
```

### Branch Naming

- `feature/vehicle-sync` — new functionality
- `fix/udp-packet-drop` — bug fix
- `refactor/session-manager` — code restructure
- `docs/update-protocol` — documentation only
- `hotfix/v0.2.1-crash` — urgent fix for a release

### Commit Messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**Types:**
| Type | Use |
|------|-----|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation only |
| `refactor` | Code change that neither fixes a bug nor adds a feature |
| `test` | Adding or correcting tests |
| `perf` | Performance improvement |
| `chore` | Build process, CI, dependency updates |

**Scopes:** `server`, `client`, `protocol`, `docs`, `ci`

**Examples:**
```
feat(server): implement TCP handshake and auth flow
fix(client): prevent crash when UDP socket fails to bind
docs(protocol): add bandwidth estimation for 20-player server
refactor(server): extract session manager into separate module
test(server): add unit tests for packet serialization
```

### Pull Requests

- Every PR must target `develop` (except hotfixes → `main`)
- Fill out the PR template
- Keep PRs focused — one feature or fix per PR
- Update VERSION_PLAN.md in the PR
- Ensure the build passes before requesting review

## Code Quality Rules

### General

1. **No dead code.** Remove commented-out code, unused imports, and unreachable branches.
2. **No magic numbers.** Use named constants for protocol values, timeouts, limits, buffer sizes.
3. **No hardcoded paths.** Use config values or relative paths.
4. **Validate at boundaries.** Validate all external input (network packets, config files, user input). Trust internal function calls.
5. **Fail fast, fail loudly.** Log errors with context. Don't silently swallow failures.
6. **Prefer simplicity.** Don't abstract prematurely. Write the straightforward solution first. Refactor when patterns emerge from real usage.

### Security

> **Security is non-negotiable. Every layer of HighBeam must enforce strict security practices. No shortcuts, no exceptions.**

1. **Never trust client data.** The server MUST validate every packet — check player IDs, vehicle ownership, data sizes, and ranges. Malformed data is rejected immediately with a structured log warning.
2. **Rate limit everything.** Auth attempts, chat messages, vehicle spawns, packet frequency. Exceeding limits results in temporary bans or disconnects.
3. **Hash passwords.** Use Argon2. Never store or compare plaintext passwords.
4. **Session tokens are secrets.** Generate with CSPRNG, transmit only once, hash for UDP auth.
5. **Bound all buffers.** Set maximum sizes for incoming packets, configs, mod files. Reject anything over the limit.
6. **No path traversal.** When serving files from Resources/, resolve and validate paths to prevent `../` escapes.
7. **Sandbox plugins.** Each Lua plugin runs in an isolated state. No `os.execute`, `io.popen`, or raw FFI access. Filesystem access only through `FS.*` API.
8. **No network-exposed admin surface.** The server GUI is local-only (egui desktop rendering). Admin actions are performed through the local GUI, console, or plugin API — never over the network.
9. **Validate everything at every layer.** Even internal functions that receive data originating from client input must validate. Defense in depth.

### Protocol

1. **Every packet has a version.** TCP packets include `"version": 1` in JSON. UDP packets are implicitly versioned by packet type layout.
2. **Length-prefix all TCP.** 4-byte LE uint32 length before every JSON payload. Never rely on newlines or delimiters.
3. **Authenticate UDP.** Every UDP packet starts with a 16-byte session token hash. Drop anything that doesn't match.
4. **Handle partial reads.** TCP is a stream — always buffer and frame correctly.
5. **Document new packet types** in PROTOCOL.md before implementing them.

## Testing Strategy

### Server (Rust)

```bash
# Run all tests
cargo test

# Run a specific test
cargo test test_packet_serialization

# Run with logging output
RUST_LOG=debug cargo test -- --nocapture
```

- **Unit tests** for: packet serialization/deserialization, auth logic, config parsing, state management
- **Integration tests** for: full connection handshake, vehicle spawn/relay cycle, plugin event firing

### Client (Lua)

Client testing is manual within BeamNG.drive:
1. Drop the mod zip into BeamNG's mods folder
2. Launch BeamNG.drive
3. Open the HighBeam connection UI
4. Connect to a running test server
5. Verify behavior in the BeamNG console (press `~` to open)

## Common Patterns

### Adding a New Packet Type

1. Read `docs/architecture/PROTOCOL.md`
2. Add the packet type to the protocol doc first (document before coding)
3. Server: Add variant to the packet enum in `server/src/net/packet.rs`
4. Server: Implement serialize/deserialize with tests
5. Server: Add handler in the appropriate module
6. Client: Add encode/decode in `client/lua/ge/extensions/highbeam/protocol.lua`
7. Client: Add handler
8. Update VERSION_PLAN.md

### Adding a Plugin API Function

1. Read `docs/architecture/SERVER.md` — Plugin Runtime section
2. Add the function to `server/src/plugin/api.rs`
3. Document it in SERVER.md under the HB.* API table
4. Write a test plugin that exercises it
5. Update VERSION_PLAN.md

### Adding a New Config Option

1. Add the field to `ServerConfig.toml` with a sensible default
2. Add parsing in `server/src/config.rs`
3. Document in SERVER.md under Configuration
4. Update VERSION_PLAN.md
