# PR Task List: Community Node UX & Port Hardening (v0.8.0-dev.4)

## Overview
This PR resolves critical blocker (0.0.0.0 advertisement), improves operator UX, and aligns port exposure with security best practices. Fixes enable the community node discovery mesh to function on public internet and improves operator guidance end-to-end.

---

## Phase 1: Critical Blocker — Public Address Configuration

### Task 1.1: Add PublicAddr to server config
- **File:** `server/src/config.rs`
- **Change:** Add `pub_addr: Option<String>` to `GeneralConfig` struct
- **Values:** None by default; operator sets to public IP or domain when joining mesh
- **Why:** `/resolve/{id}` currently returns 0.0.0.0 (unroutable); must use public address

### Task 1.2: Update config defaults file
- **File:** `server/ServerConfig.default.toml`
- **Change:** Add commented example:
  ```toml
  # PublicAddr = "your.public.ip.or.domain"  # Required only for community node participation
  ```
- **Why:** First-time operators see the option and understand when to set it

### Task 1.3: Fix community node self-entry advertisement
- **File:** `server/src/community_node.rs` — function `update_self_entry()`
- **Change:** 
  - If `config.general.pub_addr` is set and valid → use it as game/node broadcast address
  - If not set → do NOT add self to servers list; log warning "Community node disabled for lack of PublicAddr"
  - Format: `{pub_addr}:{port}` for game addr, `{pub_addr}:{listen_port}` for node addr
- **Why:** Ensures resolve returns a routable address or the server doesn't advertise at all

### Task 1.4: Add validation for PublicAddr
- **File:** `server/src/validation.rs`
- **New function:** `validate_public_address(addr: &str) -> Result<()>`
- **Rules:**
  - Must not be empty when provided
  - Must not be 0.0.0.0, 127.x.x.x, 192.168.x.x, 10.x.x.x, 172.16–31.x.x (private/loopback)
  - Max 253 chars (DNS limit)
  - Can be IP or hostname
- **Where used:** Apply on server startup and when operator changes it via GUI

### Task 1.5: GUI feedback for missing PublicAddr
- **File:** `server/src/gui.rs` — function `render_community()`
- **Change:** 
  - If enabled and no PublicAddr → show yellow warning: "⚠ PublicAddr not configured — server will not be listed in the mesh"
  - Add hint next to status: "Set your public IP or domain in ServerConfig.toml and restart to participate"
- **Why:** Operator immediately understands why their server isn't appearing in browse lists

## Phase 2: User Experience - Guidance & Clarity

### Task 2.1: GUI Community tab — Add introductory text
- **File:** `server/src/gui.rs` — `render_community()`

- **Text:** "Community Node Discovery allows your server to be found by players without them knowing your IP. Enable to join the mesh, then add at least one seed node (another community server). Your server will gossip with peers every 30 seconds to keep discovery up-to-date."

- **File:** `server/src/gui.rs` — HTTP Port row

## Phase 3: Port Hardening & Documentation

### Task 3.1: Document current port exposure
- **File:** `docs/architecture/SERVER.md`
- **Location:** After "Minimize attack surface" paragraph
- **Add:** New section "Network Ports — Minimal to Maximal Configuration"
  - **Minimal (default):** 18860 TCP+UDP only → gameplay only
  - **With mod sync:** 18860 + 18861 TCP → add launcher mod transfers
  - **With mesh:** 18860 + 18862 TCP → add community discovery gossip (18860 + 18861 + 18862 if all enabled)
  - **Local only:** Launcher IPC on loopback ephemeral (127.0.0.1:0) — not internet-exposed
- **Why:** Explicitly maps when ports appear, reducing operator confusion

### Task 3.2: Update Dockerfile to reflect community node
- **File:** `server/Dockerfile`
- **Location:** After existing EXPOSE directives
- **Add comment:** Community node HTTP port added only if enabled in config
- **Keep as-is for now:** Expose 18862 always; operator can disable container exposure if node is disabled
- **Why:** Document the expectation

### Task 3.3: Update docker-compose comments
- **File:** `server/docker-compose.yml`
- **Add:** Comment for each port explaining when it's needed
  - 18860: Always (gameplay)
  - 18861: Optional (mod sync, defaults to Port+1)
  - 18862: Optional (community node mesh, only if enabled in config)
- **Why:** Deployment operators see clear reasoning for each port mapping

### Task 3.4: Update README firewall section
- **File:** `README.md` — Firewall section
- **Change:** Expand to include optional ports and when needed:
  ```
  Required: 18860 (TCP + UDP) — gameplay
  Optional: 18861 (TCP) — if using remote launcher mod sync
  Optional: 18862 (TCP) — if joining community mesh
  ```
- **Why:** Users know upfront what might be needed

### Task 3.5: Stage and commit Cargo.lock files
- **Files:** `server/Cargo.lock`, `launcher/Cargo.lock`
- **Changes:** Both already have correct dev.4 versions in working tree
- **Action:** `git add server/Cargo.lock launcher/Cargo.lock && git commit -m "chore(deps): stage Cargo.lock at v0.8.0-dev.4"`
- **Why:** Completes version alignment across all config files
